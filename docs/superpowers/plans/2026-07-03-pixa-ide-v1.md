# Pixa IDE v1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the pixa-agent VS Code extension (agent chat, provider layer, tools, indexer, multi-file change-set review) plus the Pixa IDE fork pipeline, per spec `docs/superpowers/specs/2026-07-03-pixa-ide-design.md`.

**Architecture:** Monorepo. `packages/pixa-agent` is a VS Code extension: a webview chat panel drives an agent loop that calls a provider-agnostic model layer (OpenRouter first) with native function calling; tools mutate files only through a staged ChangeSet with diff review; terminal/git-commit require user approval via an ApprovalService bridged to the webview. `ide/` holds the VS Code OSS branding/build pipeline.

**Tech Stack:** TypeScript, esbuild (extension bundle), vitest (unit tests for host-independent modules), VS Code Extension API, OpenRouter chat/completions (OpenAI-compatible, SSE), `@vscode/ripgrep` binary bundled with VS Code.

## Global Constraints

- All file paths the agent touches must resolve inside the workspace root; reject escapes (spec §2.3).
- File mutations NEVER write disk directly — always staged via ChangeSet (spec §2.4).
- `run_command` and `git_commit` require explicit user approval (spec §3).
- No concrete provider referenced outside `providers/` (spec §2.1).
- Models defined only in `models.json`; default model id `glm` with slug `z-ai/glm-5.2` and free fallback entry (spec §2.1).
- API keys only in `SecretStorage` (spec §3).
- Token budget heuristic: chars/4 (spec §2.5).
- Agent loop max 30 iterations, AbortSignal cancellation (spec §2.2, §4).

```ts
// providers/types.ts
export type Role = "system" | "user" | "assistant" | "tool";
export interface ToolCall { id: string; name: string; arguments: string } // raw JSON string
export interface ChatMessage {
  role: Role; content: string;
  toolCalls?: ToolCall[];        // assistant messages
  toolCallId?: string;           // tool result messages
}
export interface ToolSchema { name: string; description: string; parameters: object } // JSON Schema
export interface ChatRequest { model: string; messages: ChatMessage[]; tools: ToolSchema[]; temperature?: number }
export interface StreamDelta { text?: string }
export interface ChatResult { content: string; toolCalls: ToolCall[]; finishReason: string }
export interface ModelProvider {
  id: string;
  chat(req: ChatRequest, onDelta: (d: StreamDelta) => void, signal: AbortSignal): Promise<ChatResult>;
}
export interface ModelEntry { id: string; label: string; provider: string; slug: string; contextWindow: number; supportsTools: boolean }
```

- `registry.ts`: `loadModels(json: string): ModelEntry[]` (validates fields, throws on malformed); `class ProviderRegistry { register(p: ModelProvider); resolve(modelId: string, models: ModelEntry[]): { provider: ModelProvider; entry: ModelEntry } }`.
- `openrouter.ts`: `class OpenRouterProvider implements ModelProvider` — POST `https://openrouter.ai/api/v1/chat/completions`, `stream: true`, maps ChatMessage↔OpenAI format (tool_calls, tool role), parses SSE lines (`data: {...}` / `data: [DONE]`), accumulates streamed tool-call argument fragments by index, returns ChatResult. Constructor takes `getApiKey: () => Promise<string|undefined>`. Extract pure function `parseSseChunk(buffer: string): { events: string[]; rest: string }` and `accumulateToolCallDelta(acc, delta)` for unit testing.
- `models.json` entries (id/label/provider/slug/contextWindow/supportsTools): `glm`→`z-ai/glm-5.2` 200000 true; `glm-free`→`z-ai/glm-4.5-air:free` 128000 true; `qwen`→`qwen/qwen3-coder` 262144 true; `deepseek`→`deepseek/deepseek-chat-v3.1` 163840 true; `claude`→`anthropic/claude-sonnet-5` 1000000 true; `gpt`→`openai/gpt-5.2` 400000 true; `gemini`→`google/gemini-3-pro` 1048576 true.

**Interfaces (Produces):**

```ts
// tools/types.ts
export interface ToolContext {
  workspaceRoot: string;
  changeSet: ChangeSet;
  index: RepoIndex;                     // Task 5
  approvals: ApprovalService;           // defined here, implemented by UI (Task 7)
  readWorkspaceFile(absPath: string): Promise<string | null>;
  emit(event: AgentEvent): void;        // AgentEvent defined in Task 6
}
export interface ApprovalService { requestApproval(kind: "command" | "commit", detail: string): Promise<boolean> }
export interface Tool { schema: ToolSchema; execute(args: any, ctx: ToolContext): Promise<string> }
export class ToolRegistry { register(t: Tool): void; schemas(): ToolSchema[]; run(name: string, argsJson: string, ctx: ToolContext): Promise<string> }
// registry.run: JSON.parse args (malformed → return "Error: invalid arguments ..."), unknown tool → error string; tool throw → "Error: <message>". Never rejects.
```

- `paths.ts`: `resolveInWorkspace(root: string, rel: string): string` — normalize, join, verify `path.relative(root, abs)` doesn't start with `..` and isn't absolute; throws `Error("Path escapes workspace")`. Pure, unit-tested (incl. `..\\..` and absolute Windows paths).
- `fs.ts` tools: `read_file{path,offset?,limit?}` (reads pending ChangeSet content if staged, else disk; numbered lines; cap 2000 lines), `edit_file{path,old_string,new_string}` → ChangeSet.stageEdit, `write_file{path,content}` → stageWrite (reads original if file exists), `create_file{path,content?}` → stageWrite(original null), `list_directory{path}`.
- `search.ts`: `search_workspace{regex,glob?}` spawns ripgrep at `${vscode.env.appRoot}/node_modules/@vscode/ripgrep/bin/rg` (fallback `rg` on PATH), `-n --max-count 50`, returns matches; `get_project_map{}` and `get_file_outline{path}` delegate to `ctx.index`.
- `terminal.ts`: `run_command{command,cwd?}` → `ctx.approvals.requestApproval("command", command)`; denied → `"User declined to run this command."`; approved → `child_process.exec` (shell, cwd=workspace, timeout 120s, maxBuffer 1MB) → return exit code + stdout/stderr (truncate 8k chars).
- `git.ts`: `git_status{}`/`git_diff{path?}` run git directly (read-only, no approval); `git_commit{message}` → approval("commit", message) then `git add -A && git commit -m <msg>`.
- `registry.ts` also exports `registerBuiltinTools(reg: ToolRegistry)`.

**Steps:**
- [ ] Failing tests for `resolveInWorkspace` (valid rel, `..` escape, absolute path, `.` root).
- [ ] Implement paths.ts → tests PASS.
- [ ] Implement types/registry/fs/search/terminal/git tools. `tsc --noEmit` clean.

### Task 6: Agent loop, context manager, system prompt

**Files:**
- Create: `packages/pixa-agent/src/agent/systemPrompt.ts`, `src/agent/contextManager.ts`, `src/agent/loop.ts`, `src/agent/events.ts`
- Test: `test/contextManager.test.ts`

**Interfaces (Produces):**

```ts
// agent/events.ts — webview protocol shares these
export type AgentEvent =
  | { type: "assistant-delta"; text: string }
  | { type: "assistant-done" }
  | { type: "tool-start"; callId: string; name: string; summary: string }
  | { type: "tool-end"; callId: string; result: string }
  | { type: "changeset-updated"; files: { path: string; status: string }[] }
  | { type: "approval-request"; requestId: string; kind: "command" | "commit"; detail: string }
  | { type: "status"; text: string }
  | { type: "error"; message: string };

// agent/loop.ts
export class AgentLoop {
  constructor(deps: { registry: ProviderRegistry; tools: ToolRegistry; models: ModelEntry[]; ctx: ToolContext });
  run(userMessage: string, modelId: string, signal: AbortSignal): Promise<void>; // emits via ctx.emit
  readonly history: ChatMessage[];
  reset(): void;
}
```

- `systemPrompt.ts`: `buildSystemPrompt(info: { workspaceName: string; os: string; projectMap: string }): string` — expert coding agent persona; rules: never assume file contents (use tools), prefer small targeted `edit_file` changes, verify with search/read before editing, all edits are staged for user review, commands require user approval, finish with a concise summary of changes.
- `contextManager.ts` (pure): `estimateTokens(s: string): number` (ceil chars/4); `pruneHistory(messages: ChatMessage[], budgetTokens: number): ChatMessage[]` — keep system + last user message always; walking oldest-first: first replace oldest oversized tool results with `"[result truncated]"`, then drop oldest turn-pairs, until under budget.
- `loop.ts`: run(): push user msg; loop ≤30: build request `[system, ...pruned]` with `budget = entry.contextWindow − 8000`; `provider.chat(...)` streaming deltas → emit; if toolCalls: push assistant msg w/ toolCalls, for each call emit tool-start, `tools.run(...)`, push `{role:"tool", toolCallId, content: result}`, emit tool-end + changeset-updated; else push assistant, emit assistant-done, break. Catch abort → status "Stopped"; other errors → emit error.

**Steps:**
- [ ] Failing tests: estimateTokens; pruneHistory keeps system+last-user, truncates oldest tool result first, drops oldest turns, respects budget.
- [ ] `npx vitest run` → FAIL. Implement contextManager → PASS.
- [ ] Implement events/systemPrompt/loop. `tsc --noEmit` clean.
